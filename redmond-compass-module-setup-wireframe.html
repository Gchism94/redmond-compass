Annotations

The numbered shell is shared by all three; field sets differ.

  1. Wizard header + step rail

    Short, skippable wizard — 3 steps max (identity → essentials → preview/go-live). Sensible defaults so a tool can go live in one pass; advanced settings collapse out of the way.

  2. Module-specific fields

    The only part that changes. Bookings = services, availability, buffers, approval. Inquiry = form fields, routing, auto-reply, and the private-feedback toggle (the anti-Yelp routing). Loyalty = program type, earn rule, reward, and the “show progress” toggle that powers the consumer wallet.

  3. Live preview

    Shows the exact button (Book / Request a quote / Join rewards) as it’ll render on S5, so setup and result stay tied. Same preview slot for every module.

  4. Manage destination

    Setup is one-time; day-to-day lives in the Dashboard (B1) — calendar, inbox, or ledger. The active-modules strip on B1 surfaces counts and deep-links here.

  5. Sticky save bar

    Save draft / Go live, always reachable. Shared rules: a module can be paused later without losing data, and renders nothing on the public profile until it’s live.

Shared module contract

  • Pro-gated — Free/Member tapping “set up” routes to B6 first.
  • Three surfaces each: setup (here) · manage (B1) · public face (S5).
  • Pause without data loss; renders nothing on a non-live profile.
  • Pushes a count to the B1 active-modules strip + notifies on new activity.
  • Loyalty alone adds a consumer surface — the Rewards wallet in Saved (S7).

Open questions. Whether Bookings needs calendar-sync in v1 or can launch as in-app-only · how much of the Inquiry form is fixed vs owner-editable · whether Loyalty redeem is staff-scan, owner-tap, or both · default on/off for the private-feedback routing toggle. Next candidates: the B1 Dashboard wireframe (where these are managed and where membership is pitched), or the consumer-side module surfaces on S5 and the Rewards wallet in S7.
